Problem & use case

Treat a full year of weather observations as a continuous stream and derive analytics in real time — daily averages, frost/blizzard/wet-period detection, forecasting, rankings, tourism scoring, marine routing, and ML clustering.

Why historical 2025 data?

There is no live 2026 feed for our use case, so we replay the historical 2025 archive — using each record's own observation timestamp (event time) so windows behave exactly as if it were live.

Why streaming (not batch)?

The pipeline is decoupled and replayable: producers, the broker, and processors can each fail and restart independently without losing data — the topic is the durable buffer between them.

Notes: Key framing: historical data replayed on event time. This lets us demonstrate real streaming semantics (windowing, out-of-order handling, resume) on a reproducible dataset.

Technology choices — with arguments

Docker & Docker Compose

Team runs 2 Macs + 2 Windows machines → containers give everyone an identical, one-command local environment.

Apache Kafka 3.8.1 (KRaft mode)

Durable, replayable commit log; no ZooKeeper needed (broker + controller in one). Decouples ingestion from processing.

Java 17 + Kafka Streams  (not Python)

We tried Python first — but it has no real / maintained Kafka Streams processing API (libraries undeveloped or deprecated). Kafka Streams is the officially supported stream-processing API, and it's on the JVM.

Custom Python FTP producer  (not Kafka Connect)

Kafka Connect's FTP source connector had licensing concerns and less flexibility. A custom script lets us track processed files and resume without duplicates.

Notes: These four decisions are the ones the instructor will ask "why". The Python→Java switch is the most important one to be able to defend.

Technology choices — supporting stack

Postgres 16

Stores/caches the aggregated results as dashboard tables; idempotent upserts keyed by station/day.

Grafana 11

One dashboard per pattern, reading straight from Postgres. Fast to build, familiar to reviewers.

MOA

Data ingestion — the FTP producer

  • Lists the FTP directory, streams each .gz (never stored on disk)
  • Decodes each line → wraps it in a JSON envelope → publishes to noaa.weather.raw
  • Message key = <file path>:<record #>
  • acks=all + retries; flushes progress every ~1000 records
  • Checkpoints to a state file in a Docker volume → resume support (next slide set)
// JSON envelope written to Kafka
{
  "schema_version": 1,
  "source": { "type":"noaa_isd_historical_ftp",
             "host":..., "path":..., "file_name":... },
  "observed_at": "2025-..T..Z",   // event time
  "ingested_at": "...",
  "record_number": 42,
  "payload_format": "isd-fixed-width",
  "payload_text": "<raw ISD line>"
}
Notes: We keep the raw fixed-width line in payload_text and parse it downstream in Java — so the producer stays dumb and the parsing logic lives with the processing. observed_at is extracted here so downstream can use event time.

Data streaming & parsing

Kafka Streams pipeline (per record):

  • Consume noaa.weather.raw, deserialize JSON (Jackson)
  • Parse fixed-width ISD → NoaaObservation: station (USAF+WBAN), date/time, temperature, wind, visibility→sky-clarity score, rain duration
  • Honor missing-value sentinels (+9999, 999999) & quality code 9
  • Fan out to whichever of the 11 patterns are enabled

Event time, not wall-clock

A custom TimestampExtractor uses each record's observed_at. Essential when replaying 2025 history — windows are anchored to when the weather actually happened, not when we replayed it.

Offsets reset = earliest, so a fresh consumer group reprocesses the whole year; commit interval = 1s.

Notes: Parsing lives in IsdRecordParser — it reads specific column ranges. The sky-clarity score is derived from visibility (ISD has no sunshine field). Event time is the single most important streaming concept in this project.

Short-term forecast via linear regression.

How
Least-squares slope over a 30-day sliding window advancing 1 day
Forecast
Project the line +24 h from the latest reading
Why 30 days
Long enough for a stable trend, but short enough to react; daily advance refreshes the forecast
Output
temperature-forecast
Dashboard
NOAA Temperature Forecast 2025
TemperatureForecastTopology.java
// Overlapping 30-day histories produce a fresh daily forecast.
observations.selectKey((k,o) -> o.stationId()).groupByKey()
  .windowedBy(thirtyDays.advanceBy(oneDay))
  // Update regression state, then extend its trend by 24 hours.
  .aggregate(TrendAggregate::new,addReading)
  .toStream().mapValues(t -> t.latestFit()+t.slope()*86400);

// Aggregate updates regression from running sums, not stored rows.
sumX += time; sumY += temp; sumXY += time*temp;
slope = (count*sumXY - sumX*sumY) / denominator;
Trend + 24 h projection
Notes: Forecast is anchored at the most recent observation, not the window average, so the projection starts from reality. Sliding window = overlapping, advances daily.

P1

Daily average temperature

  • StationDayKey combines station and date, so all readings for one station-day update the same Kafka state entry.
  • The aggregate stores only sum and count. Every new temperature increments both values.
  • averageTemperatureCelsius() derives sum / count; the KTable therefore always represents the latest daily average.

Why this matters: Kafka can restore sum and count from the changelog after a restart. It does not need the original readings or a previously rounded average to continue correctly.

DailyAverageTemperatureTopology.java
public static KTable<String, TemperatureAggregate> build(
    KStream<String, NoaaObservation> observations) {
  // One Kafka key represents one station and calendar day.
  return observations
    .filter((key, o) -> isUsableObservation(o))
    .map((key, o) -> KeyValue.pair(
        StationDayKey.fromObservation(o).asKey(), o))
    .groupByKey(Grouped.with(Serdes.String(), observationSerde))
    // The materialized KTable stores the running average state.
    .aggregate(TemperatureAggregate::new,
        (stationDay, o, avg) -> avg.add(o.temperatureCelsius()),
        Materialized.with(Serdes.String(), aggregateSerde));
}
TemperatureAggregate.add() / averageTemperatureCelsius()
// Example state for one station-day:
// before: sum=10, count=1; new value=14
count++;
sum += value;

// after: sum=24, count=2; result=12 °C
return count == 0 ? 0.0 : sum / count;
P2

Frost days per month

  • The first count() stores the number of frost readings for each station-day: the same day can change from 1 → 2 → 3.
  • groupBy(stationMonth) converts every positive daily count to a monthly contribution of exactly 1: it is still only one frost day.
  • When the daily row changes, Kafka updates the monthly total as total − old contribution + new contribution.

Why this matters: If a day changes from 1 to 2 frost readings, its monthly contribution changes from 1 → 1. Kafka calculates 10 − 1 + 1 = 10; without subtraction it would incorrectly become 11 frost days.

MonthlyFrostDaysTopology.java
public static KTable<String, Long> build(
    KStream<String, NoaaObservation> observations, int year) {
  // First collapse many sub-zero readings into one station-day.
  KTable<String, Long> frostDays = observations
    .filter((key, o) -> o.temperatureCelsius() < 0.0)
    .map((key, o) -> KeyValue.pair(stationDayKey(o), o))
    .groupByKey()
    .count();

  // Map every positive daily count to exactly one frost-day contribution.
  return frostDays
    .groupBy((day, count) ->
        KeyValue.pair(stationMonthKey(day), 1L))
    .aggregate(() -> 0L,
        (month, added, total) -> total + added,
        (month, removed, total) -> total - removed);
}
MonthlyFrostDaysTopology.build() — why subtract?
// Same day receives another frost reading:
// daily count: 1 → 2
// monthly contribution: 1 → 1

// Kafka first removes the old table value ...
(month, oldValue, total) -> total - oldValue
// ... and then adds the replacement value.
(month, newValue, total) -> total + newValue

// Example: 10 - 1 + 1 = 10 frost days.
P3

Rapid temperature change

  • selectKey(stationId) and windowedBy(...) create independent 24-hour state for every station.
  • The first reading initializes lastTemperature and lastObservedAt. Each later reading is compared with that previous event.
  • The temperature difference is divided by the exact elapsed hours, producing comparable °C-per-hour rates even with irregular sampling.

Why this matters: The pattern measures speed of change, not just temperature difference. A 6 °C change in one hour is more rapid than the same change over six hours.

RapidTemperatureChangeTopology.java
public static KStream<String, RapidTemperatureChangeEvent> build(
    KStream<String, NoaaObservation> observations, int windowHours) {
  TimeWindows window =
      TimeWindows.ofSizeAndGrace(Duration.ofHours(windowHours), grace);

  // Kafka keeps one rate-of-change aggregate per station/window.
  return observations
    .selectKey((key, o) -> o.stationId())
    .groupByKey()
    .windowedBy(window)
    .aggregate(RapidTemperatureChangeAggregate::new,
        (station, o, rates) -> rates.add(o))
    .toStream()
    .map((windowedStation, rates) -> toEvent(windowedStation, rates));
}
RapidTemperatureChangeAggregate.add()
// The first event only initializes the comparison state.
if (lastObservedAt != null) {
  deltaTemp = currentTemp - lastTemperature;
  hours = Duration.between(lastObservedAt, currentTime)
      .toMillis() / 3_600_000.0;
  rate = deltaTemp / hours;

  minRate = Math.min(minRate, rate);
  maxRate = Math.max(maxRate, rate);
  sumRate += rate; rateCount++;
}
// Current becomes the reference for the next event.
lastTemperature = currentTemp;
lastObservedAt = currentTime;
P4

Tourism weather quality index

  • regions.resolve(stationId) maps stations to a tourism region; re-keying makes all stations in that region share one daily window.
  • The aggregate keeps sums and a count, then calculates regional averages for temperature, wind, and sky clarity.
  • Each average becomes a 0–100 comfort score. The final index weights temperature 55%, wind 25%, and clarity 20%.

Why this matters: The result compares regions on one consistent scale instead of comparing raw values with different units such as °C, m/s, and percent clarity.

TourismWeatherQualityTopology.java
public static KStream<String, TourismWeatherQualityEvent> build(
    KStream<String, NoaaObservation> observations,
    StationRegionResolver regions) {
  // Re-key stations to regions before the daily aggregation.
  return observations
    .map((key, o) -> KeyValue.pair(
        regions.resolve(o.stationId()).id(), o))
    .groupByKey()
    .windowedBy(dailyWindow)
    .aggregate(TourismWeatherQualityAggregate::new,
        (region, o, score) -> score.add(o, regions.resolve(o.stationId())))
    .toStream()
    .map((windowedRegion, score) -> toEvent(windowedRegion, score));
}
TourismWeatherQualityAggregate.qualityScore()
// Derive regional daily averages from running sums.
avgTemp = temperatureSum / observationCount;
avgWind = windSpeedSum / observationCount;
avgClarity = skyClaritySum / observationCount;

// Convert different units to comparable 0–100 scores.
tempScore = Math.max(0, 100 - Math.abs(avgTemp - 22) * 6);
windScore = Math.max(0, 100 - avgWind * 3.5);

// 22 °C, low wind and clear sky produce a high index.
quality = tempScore * .55
        + windScore * .25
        + avgClarity * .20;
P5

Average rain duration

  • The filters keep only the selected year and observations with rainDurationHours > 0; dry or missing values do not contribute.
  • StationYearKey routes all rainy observations for one station and year to the same materialized state.
  • The aggregate stores total rain-duration hours and event count. Their division is the average duration of a reported rain period.

Why this matters: This is not “rain hours divided by every weather reading.” It averages only observations that report rain duration, and Kafka can resume from total and count after a restart.

YearlyRainDurationTopology.java
public static KTable<String, RainDurationAggregate> build(
    KStream<String, NoaaObservation> observations, int year) {
  // Station-year is the key for the recoverable running state.
  return observations
    .filter((key, o) -> o.rainDurationHours() != null

Top-10 hottest & coldest stations

  • Kafka first keeps exactly one summary per station. Math.min and Math.max update that station’s lowest and highest temperature seen during the year.
  • The same station summary is sent to two lists: coldest compares its minimum, while hottest compares its maximum.
  • Before reinserting a changed station, its previous list entry is removed. Each station therefore appears at most once, with its current annual extreme.

Why this matters: The result is the ten stations with the lowest annual minima and the ten stations with the highest annual maxima—not the ten latest readings. Each update only searches a list of at most ten candidates.

Temperature trend forecasting

  • A 30-day window with advanceBy(1 day) creates a new overlapping forecast window each day for every station.
  • The aggregate stores five least-squares sums instead of all readings. From them, getSlope() calculates the temperature trend in °C per second.
  • The fitted value at the latest observation is treated as “now”; adding slope × 86,400 projects that trend 24 hours forward.

Why this matters: Anchoring at the latest fitted point avoids forecasting 24 hours from the window’s old average. The model uses event time and bounded regression state.

TemperatureForecastTopology.java
public static KStream<String, TemperatureForecastEvent> build(
    KStream<String, NoaaObservation> observations) {
  TimeWindows window = TimeWindows
      .ofSizeAndGrace(Duration.ofDays(30), grace)
      .advanceBy(Duration.ofDays(1));

  // Each overlapping window maintains its own regression sums.
  return observations
    .selectKey((key, o) -> o.stationId())
    .groupByKey()
    .windowedBy(window)
    .aggregate(TemperatureTrendAggregate::new,
        (station, o, trend) -> trend.add(o))
    .toStream()
    .map((stationWindow, trend) -> toForecast(stationWindow, trend));
}
TemperatureTrendAggregate + toForecast()
// Use time relative to the first event for stable numbers.
x = observedAt.toEpochSecond() - firstTimestamp;
count++;
sumX += x; sumY += temperature;
sumXY += x * temperature;
sumX2 += x * x;

// Least-squares slope: temperature change per second.
slope = (count * sumXY - sumX * sumY)
      / (count * sumX2 - sumX * sumX);

// Start at the latest fitted value, then move ahead 24h.
latestFit = intercept + slope * lastRelativeX;
forecastNext24h = latestFit + slope * 86_400;

Caribbean maritime routing

  • Both streams are re-keyed by seaAreaId, so an AIS position can only meet buoy observations from the same sea area.
  • The stream–stream join additionally requires their event timestamps to differ by no more than the configured join window.
  • For each valid pair, wave height and wind speed produce a bounded risk score, risk class, route advice, and an adjusted ETA.

Why this matters: Matching only by area could combine a current ship position with stale weather. The window makes the recommendation agree in both location and time.

MaritimeRoutingTopology.java
public static KStream<String, RouteRecommendationEvent> build(
    KStream<String, AisPositionEvent> ais,
    KStream<String, BuoyObservationEvent> buoys) {
  // Both streams need the same sea-area key for the join.
  KStream<String, AisPositionEvent> keyedAis =
      ais.selectKey((key, event) -> event.seaAreaId());
  KStream<String, BuoyObservationEvent> keyedBuoys =
      buoys.selectKey((key, event) -> event.seaAreaId());

  // Match ship and buoy events within the event-time interval.
  return keyedAis.join(keyedBuoys,
      (ship, buoy) -> recommendation(ship, buoy, joinWindow),
      JoinWindows.ofTimeDifferenceWithNoGrace(joinWindow));
}

Wet period detection

  • selectKey(stationId) gives each station one independent state machine stored in a persistent Kafka key-value store.
  • The first wet reading creates state with start time and count. Further wet readings increment the count and emit nothing.
  • The first following dry reading calculates duration, deletes the open state, and emits exactly one completed WetPeriodEvent.

Why this matters: Output is produced per complete wet period, not per rainy reading. Changelog-backed state preserves an unfinished period across processor restarts.

WetDryPeriodTopology.java
public static KStream<String, WetPeriodEvent> build(

        "wet-dry-period-state")
    // The transformer returns null until a wet period closes.
    .filter((station, event) -> event != null);
}

Blizzard detection

  • windowedBy(...advanceBy...) maintains overlapping 24-hour evidence state independently for each station.
  • Every reading may increment the freezing, strong-wind, or precipitation counter. The three signals may come from different readings inside the same window.
  • isDetected() requires all three counters to be positive; cold and wind without precipitation are not classified as a blizzard.

Why this matters: NOAA provides rain duration rather than an explicit snow signal. During freezing conditions it serves as a precipitation proxy, so this is a detection of blizzard-like conditions.

BlizzardDetectionTopology.java
public static KStream<String, BlizzardEvent> build(
    KStream<String, NoaaObservation> observations) {
  TimeWindows window = TimeWindows
      .ofSizeAndGrace(Duration.ofHours(24), grace)
      .advanceBy(Duration.ofHours(advanceHours));

  // Aggregate threshold evidence inside each station/window.
  return observations
    .selectKey((key, o) -> o.stationId())
    .groupByKey()
    .windowedBy(window)
    .aggregate(BlizzardWindowAggregate::new, addEvidence)
    .toStream()
    .map((stationWindow, evidence) -> toEvent(stationWindow, evidence))
    .filter((key, event) -> event.isDetected());
}

P11

Weather regime clustering (MOA)

  • The first aggregate reduces all readings of one station-day to six features: average and range of temperature, average and maximum wind, clarity, and rain duration.
  • normalized() maps every feature to 0–1, preventing large-unit values from dominating Euclidean distance.
  • assignAndLearn(...) updates CluStream with the daily vector, finds its nearest micro-cluster, and maps that center to a readable weather-regime label.

Why this matters: Kafka emits one meaningful daily vector instead of clustering every raw reading. The online model adapts one vector at a time without storing and retraining the full history.

WeatherRegimeTopology.java
public static KStream<String, WeatherRegimeEvent> build(
    KStream<String, NoaaObservation> observations) {
  OnlineWeatherRegimeClusterer model =
      new OnlineWeatherRegimeClusterer(clusterCount, horizon);

  // First produce one feature vector per station and day.
  return observations
    .map((key, o) -> KeyValue.pair(stationDayKey(o), o))
    .groupByKey()
    .aggregate(StationDayWeatherAggregate::new,
        (day, o, features) -> features.add(o))
    .toStream()
    .filter((day, features) -> features.hasEnoughSignal())
    // Assign the vector and immediately update the online model.
    .mapValues((day, features) ->
        toEvent(day, model.assignAndLearn(features.toFeatureVector())));
}

Challenges

Python had no usable Streams API

Libraries deprecated/undeveloped → migrated processing to Java Kafka Streams.

Resource pressure

Running all 11 patterns spiked disk & CPU → added STREAM_PATTERN to run one at a time (with --force-recreate).

Large datasets

1.5 GB / ~12.8 k files + a big AIS archive → configurable caps for local runs.

Fixed-width ISD parsing

Column offsets, quality codes, missing-value sentinels — easy to get subtly wrong.

Missing dimensions

No sunshine (→ visibility proxy) and no marine data (→ separate producer) in ISD.

Replay vs resume

Managing the Kafka Streams application.id to control when we resume vs reprocess.

Notes: The application.id challenge is the bridge into the fault-tolerance section — the ID is derived deterministically from the pattern config so the same config resumes and a changed config replays.

What happens if a service stops?

Producer stops

Keeps a JSON checkpoint in a Docker volume (producer_state): which files are done + records sent per file. On restart it skips completed files (matched by size/modified) and resumes a partial file after the last confirmed record. No loss, no duplicate re-send. (Marine producer tracks completed dates.)

Kafka broker stops

The log is durable on disk (kafka_data) and the raw topic has infinite retention (retention.ms=-1). Nothing is lost; on restart every message is still there. Producer uses acks=all + retries.

Stream processor (pattern) stops

Kafka Streams commits consumer offsets (every ~1 s). On restart with the same application.id it resumes from the last committed offset — it does not restart from zero.

Notes: Three independent failure domains, each with its own recovery mechanism. The theme: state lives in durable stores (volume, Kafka log, offsets), never only in memory.

Is data lost? Can we resume at the last row?

No data is lost — and yes, it can be re-read and continued at the last processed record.

Why nothing is lost

Kafka is a durable, replayable log; the raw topic has infinite retention. Any consumer can re-read from any offset, at any time.

Continue at the "last row"

With the same application.id, the stream continues from the last committed offset — the offset is the bookmark for the last row processed.

auto.offset.reset = earliest: a fresh application.id (new consumer group) intentionally replays the entire 2025 year — useful to rebuild dashboards from scratch. The ID is generated deterministically from the pattern config (hash of pattern + year + window params): same config → resume · changed config → full replay.

Notes: This is the crux. Two behaviors, both intentional: resume (same ID) vs replay (new ID). Because retention is infinite, replay is always possible. The deterministic ID means you don't accidentally replay just by restarting.

Where does the last state come from?

A layered state model — each layer recovers from a durable source:

1 · Producer progress

JSON checkpoint file in the producer_state volume — which files & which record number.

2 · Consumer position

Committed offsets in Kafka's internal __consumer_offsets topic, keyed by application.id.

3 · Operator / aggregation state

Windows, the wet/dry state machine, rankings, forecasts, clustering → local RocksDB state stores backed by Kafka changelog topics. On restart the store is restored from its changelog, so aggregations resume where they left off.

4 · Dashboard layer (Postgres)

Idempotent upserts keyed by station/day/month (daily avg, rain, frost); event tables are rebuilt from the replay. Honest caveat: the raw/parsed counters are simple increments, so a full replay under a fresh ID inflates those counters unless the ID resumes.

Notes: The answer to "where does state come from" is layered: producer file, offsets topic, changelog-backed state stores, and Postgres. The changelog restore is the classic Kafka Streams fault-tolerance mechanism. Be upfront about the counter caveat — shows we understand idempotency.

Lessons learned

Pick the ecosystem, not just the language

Kafka Streams' first-class support on the JVM outweighed our initial Python preference.

Durability is designed, not assumed

Volumes + infinite retention + offsets + changelogs together give end-to-end recovery.

Event time is everything

Replaying history correctly is only possible because windows use observed_at.

Idempotency vs replay

Upserts survive re-processing; naive counters don't — design outputs for replay.

Model honestly

Visibility-as-clarity and sea-area join keys are documented approximations, not fakes.

Resource-aware ops

One toggle (STREAM_PATTERN) made the whole thing runnable on laptops.

Notes: Tie back to the fault-tolerance story — the recovery guarantees are the strongest part of the project.
