Problem & use case

Treat a full year of weather observations as a continuous stream and derive analytics in real time — daily averages, frost/blizzard/wet-period detection, forecasting, rankings, tourism scoring, marine routing, and ML clustering.

Why historical 2025 data?

There is no live 2026 feed for our use case, so we replay the historical 2025 archive — using each record's own observation timestamp (event time) so windows behave exactly as if it were live.

Why streaming (not batch)?

The pipeline is decoupled and replayable: producers, the broker, and processors can each fail and restart independently without losing data — the topic is the durable buffer between them.

Notes: Key framing: historical data replayed on event time. This lets us demonstrate real streaming semantics (windowing, out-of-order handling, resume) on a reproducible dataset.

Technology choices — with arguments

Docker & Docker Compose

Team runs 2 Macs + 2 Windows machines → containers give everyone an identical, one-command local environment.

Apache Kafka 3.8.1 (KRaft mode)

Durable, replayable commit log; no ZooKeeper needed (broker + controller in one). Decouples ingestion from processing.

Java 17 + Kafka Streams  (not Python)

We tried Python first — but it has no real / maintained Kafka Streams processing API (libraries undeveloped or deprecated). Kafka Streams is the officially supported stream-processing API, and it's on the JVM.

Custom Python FTP producer  (not Kafka Connect)

Kafka Connect's FTP source connector had licensing concerns and less flexibility. A custom script lets us track processed files and resume without duplicates.

Notes: These four decisions are the ones the instructor will ask "why". The Python→Java switch is the most important one to be able to defend.

Postgres 16

Data ingestion — the FTP producer

  • Lists the FTP directory, streams each .gz (never stored on disk)
  • Decodes each line → wraps it in a JSON envelope → publishes to noaa.weather.raw
  • Message key = <file path>:<record #>
  • acks=all + retries; flushes progress every ~1000 records
  • Checkpoints to a state file in a Docker volume → resume support (next slide set)
// JSON envelope written to Kafka
{
  "schema_version": 1,
  "source": { "type":"noaa_isd_historical_ftp",
             "host":..., "path":..., "file_name":... },
  "observed_at": "2025-..T..Z",   // event time
  "ingested_at": "...",
  "record_number": 42,
  "payload_format": "isd-fixed-width",
  "payload_text": "<raw ISD line>"
}
Notes: We keep the raw fixed-width line in payload_text and parse it downstream in Java — so the producer stays dumb and the parsing logic lives with the processing. observed_at is extracted here so downstream can use event time.

Data streaming & parsing

Kafka Streams pipeline (per record):

  • Consume noaa.weather.raw, deserialize JSON (Jackson)
  • Parse fixed-width ISD → NoaaObservation: station (USAF+WBAN), date/time, temperature, wind, visibility→sky-clarity score, rain duration
  • Honor missing-value sentinels (+9999, 999999) & quality code 9
  • Fan out to whichever of the 11 patterns are enabled

Event time, not wall-clock

A custom TimestampExtractor uses each record's observed_at. Essential when replaying 2025 history — windows are anchored to when the weather actually happened, not when we replayed it.

Offsets reset = earliest, so a fresh consumer group reprocesses the whole year; commit interval = 1s.

Notes: Parsing lives in IsdRecordParser — it reads specific column ranges. The sky-clarity score is derived from visibility (ISD has no sunshine field). Event time is the single most important streaming concept in this project.

Challenges

Python had no usable Streams API

Libraries deprecated/undeveloped → migrated processing to Java Kafka Streams.

Resource pressure

Running all 11 patterns spiked disk & CPU → added STREAM_PATTERN to run one at a time (with --force-recreate).

Large datasets

1.5 GB / ~12.8 k files + a big AIS archive → configurable caps for local runs.

Fixed-width ISD parsing

Column offsets, quality codes, missing-value sentinels — easy to get subtly wrong.

Missing dimensions

No sunshine (→ visibility proxy) and no marine data (→ separate producer) in ISD.

Replay vs resume

Managing the Kafka Streams application.id to control when we resume vs reprocess.

Notes: The application.id challenge is the bridge into the fault-tolerance section — the ID is derived deterministically from the pattern config so the same config resumes and a changed config replays.

What happens if a service stops?

Producer stops

Keeps a JSON checkpoint in a Docker volume (producer_state): which files are done + records sent per file. On restart it skips completed files (matched by size/modified) and resumes a partial file after the last confirmed record. No loss, no duplicate re-send. (Marine producer tracks completed dates.)

Kafka broker stops

The log is durable on disk (kafka_data) and the raw topic has infinite retention (retention.ms=-1). Nothing is lost; on restart every message is still there. Producer uses acks=all + retries.

Stream processor (pattern) stops

Kafka Streams commits consumer offsets (every ~1 s). On restart with the same application.id it resumes from the last committed offset — it does not restart from zero.

Notes: Three independent failure domains, each with its own recovery mechanism. The theme: state lives in durable stores (volume, Kafka log, offsets), never only in memory.

Is data lost? Can we resume at the last row?

No data is lost — and yes, it can be re-read and continued at the last processed record.

Why nothing is lost

Kafka is a durable, replayable log; the raw topic has infinite retention. Any consumer can re-read from any offset, at any time.

Continue at the "last row"

With the same application.id, the stream continues from the last committed offset — the offset is the bookmark for the last row processed.

auto.offset.reset = earliest: a fresh application.id (new consumer group) intentionally replays the entire 2025 year — useful to rebuild dashboards from scratch. The ID is generated deterministically from the pattern config (hash of pattern + year + window params): same config → resume · changed config → full replay.

Notes: This is the crux. Two behaviors, both intentional: resume (same ID) vs replay (new ID). Because retention is infinite, replay is always possible. The deterministic ID means you don't accidentally replay just by restarting.

Where does the last state come from?

A layered state model — each layer recovers from a durable source:

1 · Producer progress

JSON checkpoint file in the producer_state volume — which files & which record number.

2 · Consumer position

Committed offsets in Kafka's internal __consumer_offsets topic, keyed by application.id.

3 · Operator / aggregation state

Windows, the wet/dry state machine, rankings, forecasts, clustering → local RocksDB state stores backed by Kafka changelog topics. On restart the store is restored from its changelog, so aggregations resume where they left off.

4 · Dashboard layer (Postgres)

Idempotent upserts keyed by station/day/month (daily avg, rain, frost); event tables are rebuilt from the replay. Honest caveat: the raw/parsed counters are simple increments, so a full replay under a fresh ID inflates those counters unless the ID resumes.

Notes: The answer to "where does state come from" is layered: producer file, offsets topic, changelog-backed state stores, and Postgres. The changelog restore is the classic Kafka Streams fault-tolerance mechanism. Be upfront about the counter caveat — shows we understand idempotency.

Lessons learned

Pick the ecosystem, not just the language

Kafka Streams' first-class support on the JVM outweighed our initial Python preference.

Durability is designed, not assumed

Volumes + infinite retention + offsets + changelogs together give end-to-end recovery.

Event time is everything

Replaying history correctly is only possible because windows use observed_at.

Idempotency vs replay

Upserts survive re-processing; naive counters don't — design outputs for replay.

Model honestly

Visibility-as-clarity and sea-area join keys are documented approximations, not fakes.

Resource-aware ops

One toggle (STREAM_PATTERN) made the whole thing runnable on laptops.

Notes: Tie back to the fault-tolerance story — the recovery guarantees are the strongest part of the project.
